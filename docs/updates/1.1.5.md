# 1.1.5

日期：`2026-05-07`

## 版本概览

本轮是维护型小版本，目标是修复已确认 UI bug、审计批量性能与线程边界、补全 Console 阶段耗时，并同步文档。没有新增大功能，没有改变相似图算法、分析判断风格或修复风格。

## 相似图片组内对比窗口

- 组内对比窗口默认尺寸提升到可容纳说明区、图片预览、文件名/摘要、每图“删除此图”按钮和底部全局按钮的范围。
- 窗口尺寸按屏幕可视区收敛，不会为了默认尺寸超出屏幕。
- 图片卡片区现在是独立可滚动区域；底部“跳过本组 / 跳过所有剩余组 / 结束选择”固定在窗口底部。
- 2 张、3 张、4 张仍可在当前页打开；5 张以上继续分页，每页最多 4 张。
- 小窗口提示阈值按完整 UI 需要重新调整，提示用户图片区可滚动，按钮区仍可达。

## 性能与线程审计

- 批量分析和批量修复仍使用受控 `ThreadPoolExecutor`，结果通过 `as_completed()` 回收，并非按提交顺序串行等待。
- Tk 控件更新继续通过主线程调度，不允许后台线程直接写 Treeview、Text、Label 或 Progressbar。
- 当前主要可见开销来自 Console 每条日志重绘整块文本、进度回调高频刷新、缩略图/列表刷新、相似检测阶段读取缩略图特征、修复阶段候选评分与保存输出。
- 本轮将 Console 刷新改为合并刷新，并对分析/修复阶段进度回调做轻量节流，减少后台 worker 对 Tk 主线程的压力。
- 本轮新增批次级性能审计：记录 `total_analysis_time`、`total_wall_time`、`total_worker_time`、`average_time_per_image`、`queue/wait`、`slowest_images top 5`、`slowest_stages top 5`、`similar_detection`、`UI_update` 与 `console_flush`。
- 批量分析单张完成时改为定点更新当前 Treeview 行，不再每张完成都重建整张缩略图列表；最终完成后再做一次完整刷新，保持排序和相似组标记稳定。
- 默认自动分析 worker 上限从 8 提升到 12，并新增应用设置：自动、低、中、高、自定义 worker 数。大图批量仍默认走安全上限，避免为了占满 CPU 牺牲内存稳定性。

## GPU 加速设置

- 应用设置新增 GPU 加速：关闭 / 自动 / 开启。
- 设置面板会显示可用后端：未检测到 / CuPy CUDA / OpenCV CUDA / torch CUDA。
- CUDA、CuPy、OpenCV-CUDA、torch 都不是必需依赖；检测失败或没有设备时，程序启动和 CPU 分析不受影响，并在 Console 中说明“未启用 GPU，加速回退 CPU”。
- 当前 1.1.5 预留 GPU 数值阶段接入点，但默认分析仍走 CPU 稳定路径；不把大图对象跨设备或跨进程传递，避免引入新的内存和安装风险。

## Console 与 perf_timings

分析阶段会摘要显示：

- 读取图片
- 基础统计
- 场景判断
- 人像/清晰度/噪声/色彩判断
- cleanup candidate
- 相似图检测
- UI 刷新
- Console 刷新
- queue/wait 与 worker wall time

修复阶段会摘要显示：

- 生成修复方案
- 读取图片
- 执行主要修复步骤
- 候选评分/安全检查
- 保存输出
- 元数据保留

批量结束时会输出总耗时、平均耗时、并发模式、CPU/GPU 状态、最慢图片 top 5、最慢阶段 top 5、相似图检测耗时、UI 刷新耗时和轻量瓶颈提示。Console 不输出内部阈值、评分公式或过细算法细节。

# 2026-05-07 Real-Image Performance Addendum

- Added `/test` as the local real-photo benchmark folder. Git ignores image files under `/test`, while `test/README.md` documents that real photos are local-only and must not be committed.
- Added `benchmark_test_images.py` for repeatable `/test` runs. It safely skips when no local images exist and reports wall time, average wall time, worker cumulative time, parallel efficiency, queue/wait time, slow stages, slow images, similar detection time, issue count, and cleanup candidate count.
- Corrected Console performance wording. User summaries now lead with real `wall_time`; `worker_cumulative_time` is explicitly described as cumulative worker effort and is no longer presented as total user wait time.
- Centralized analysis worker resolution in `app_settings.resolve_analysis_worker_plan()`. Low, medium, high, and custom settings now flow to the actual `ThreadPoolExecutor` worker count and report requested vs actual workers.
- Implemented a conservative large-image working path in `analysis/core.py`: most numeric analysis runs on a bounded working image, returned dimensions and regions are scaled back to original coordinates, and noise scoring is corrected for working-image pixel scale to preserve conclusions.
- Reduced similar feature extraction cost by using Pillow JPEG `draft()` before building hashes and small feature vectors; this keeps the existing similar-image matching style.
- `/test` real-image benchmark on 16 JPGs improved high-worker wall time from 53.29s to 23.92s, while preserving 6 issue images, 3 cleanup candidates, and 4 similar groups.
- GPU remains optional detection and CPU fallback in 1.1.5; no real GPU offload is claimed yet.
# 2026-05-07 Analysis Cancel Addendum

- Tightened batch-analysis cancel state handling without changing analysis algorithms.
- Cancel now keeps the imported file list, clears this run's partial analysis results/errors/cleanup flags/similar-group markers, and re-enables controls so the user can immediately start a new analysis.
- Stale background callbacks continue to be rejected by `run_id`; canceled workers may finish CPU work, but their results are discarded and cannot write back to the UI.
- Console now records a cancel request summary with elapsed time and cleared/canceled counts, then a worker shutdown confirmation with submitted/completed/discarded/pending counts.
