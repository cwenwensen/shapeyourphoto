# 1.1.6

日期：`2026-05-08`

## 版本概览

1.1.6 是文档体系和维护规则整理版本。目标是让后续维护者能够从正式文档理解当前项目，而不是在根 README、模块索引、历史附录和旧版本记录之间反复比对。本轮不改变图像分析算法、修复风格、UI 主交互逻辑或用户数据处理规则。

## 文档体系重整

- 根 `README.md` 改为用户入口、当前能力、启动方式和正式文档阅读顺序。
- 根 `MODULES.md` 改为快速模块索引，明确当前主路径和废弃旧入口。
- `docs/README.md` 明确根文档、正式维护文档、技术专题和版本更新记录的分工。
- `docs/SYSTEM_OVERVIEW.md` 重写为当前系统主流程、关键数据结构和高敏感链路说明。
- `docs/MODULE_REFERENCE.md` 按真实模块结构更新职责，覆盖 UI、设置、扫描、分析、修复、相似图、benchmark、GPU fallback 和数据结构。
- `docs/UI_AND_WORKFLOWS.md` 统一说明导入、扫描、分析取消、修复、cleanup、相似图和 Console 工作流。
- `docs/MAINTENANCE_GUIDE.md` 重写维护规则，覆盖启动链路、Tk 主线程、run_id/cancel_event、perf 计时、EXIF Orientation、cleanup 安全删除、相似图、`test/`、GPU fallback 和文档更新。

## 新增技术专题

- `docs/technical/ANALYSIS_PIPELINE.md`：分析入口、单张阶段、working image、批量写回和相似图位置。
- `docs/technical/PERFORMANCE_AND_CONCURRENCY.md`：worker 规划、Tk 主线程、取消、Console 合并刷新、perf 规则和 GPU fallback。
- `docs/technical/SIMILAR_IMAGES.md`：相似组数据结构、检测特征、复核 UI、删除规则和禁止事项。
- `docs/technical/CLEANUP_CANDIDATES.md`：候选生成、复核、强制修复关系、安全删除和与相似图的关系。
- `docs/technical/SETTINGS_AND_SCAN.md`：应用设置、扫描模式、忽略前缀、扫描摘要和回归入口。

## 旧内容纠偏

- 明确独立 `single_image_window.py` 和“单图模式”不是当前主路径；单张图片通过主列表处理。
- 明确孤立“去噪当前”入口不是当前主路径；降噪已经并入分析和修复链。
- 明确修复完成长详情不应回退到普通 `messagebox`；当前以可筛选滚动窗口为准。
- 明确早期“设置 -> 扫描忽略目录前缀”描述属于历史背景；当前设置统一在“应用设置”面板。
- 明确相似图只作为批次附加结果，不写回单张 `AnalysisResult`。
- 明确 GPU 仅检测可选后端并回退 CPU，不是硬依赖，也不声明默认 GPU offload。
- 明确旧版本更新文档保留历史上下文；如与 1.1.6 文档冲突，以当前文档和代码为准。

## 少量残骸清理

- 清理已生成的 `__pycache__` 和 `.pyc` 缓存文件，避免临时编译产物留在工作区。
- 将 `gpu_accel.py` 中仍写作“当前 1.1.5”的状态文案改为版本无关描述，避免升级后应用内提示过时。

## 低风险基础建设补充

- `benchmark_test_images.py` 新增自动报告能力。运行后会写入被忽略的 `benchmark_reports/`，同时生成 JSON 与 Markdown 报告。
- benchmark 报告包含运行时间、图片数量、worker 模式、wall time、worker cumulative time、average wall time per image、slow stages top 5、slow images top 5、issues、cleanup candidates 和 similar groups。
- 如果存在上一份报告，会按同名模式输出 wall time、issues、cleanup candidates 和 similar groups 的简单变化。
- benchmark 可选读取本地 `test/manifest.json`，按文件名对比预期 issues、cleanup candidate、similar group 和不应出现的问题；manifest 不存在时正常跳过。
- `app_settings.py` 新增 `settings_schema_version`，当前 schema 为 `1`，并预留 `migrate_settings(old_version, data)` 入口。
- 旧版设置缺少 schema version、缺字段、字段类型错误或包含未知字段时，会被规范化为当前 schema 并保存；损坏 JSON 继续备份坏文件并回退默认值。
- `test/README.md` 补充本地样张命名和标注方式，新增可提交的 `test/manifest.example.json`；真实图片、真实 `test/manifest.json` 和 benchmark 报告继续由 `.gitignore` 排除。

## UI 主类拆分

- `ui_app.py` 收敛为主窗口装配、实例状态和高层菜单/布局 wiring，不再继续承载扫描、分析、修复、列表、Console 和复核细节。
- 新增 `ui_scan_actions.py`、`ui_analysis_actions.py`、`ui_repair_actions.py`、`ui_file_list.py`、`ui_task_console.py`、`ui_review_actions.py` 和 `ui_constants.py`，以 mixin 形式接回 `PhotoAnalyzerApp`。
- 拆分保持原有用户行为：导入、扫描、拖拽、分析取消、修复、相似图、cleanup candidate、设置、Console 和 EXIF/ICC/DPI/XMP 展示链路不变。
- 新增 `docs/technical/UI_MAIN_CLASS_SPLIT.md` 记录 UI 代码边界和后续维护规则。

## 批量修复选择语义修复

- “修复当前”继续只处理当前蓝色高亮/焦点图片。
- “批量修复勾选”改为只有真正多选行（多于 1 张）才优先使用多选集合；如果没有真正多选，则使用主列表“处理状态”勾选集合。
- 单个蓝色高亮行不再覆盖勾选集合，避免 6 张勾选目标被错误缩成 1 张当前行。
- 批量修复准备窗口在多图时显示批量说明和方法汇总，例如“已选择 6 张：包含欠曝、过曝、高调背景等不同类型，将逐张生成独立修复计划”，不再只显示焦点图自己的推荐方法。
- 批量修复仍逐图调用 repair planner / repair engine；每张图使用自己的 `AnalysisResult`、`repair_policy`、`method_ids`、`op_strengths`、候选评分和 no-op/回退决策。
- 修复完成详情新增全量目标与候选回退/no-op 统计，跳过数量按真实批量目标结果累计。

## 进度窗口与最小尺寸规则

- 分析/修复进度弹窗改为固定内部区域：标题区、说明区、进度条区、计数与耗时区、固定高度当前阶段摘要区和底部按钮区。
- 当前阶段长文本会压缩省略，详细阶段信息继续进入 Console，避免挤压进度条和底部按钮。
- 取消按钮只保留一个明确入口；重复点击会禁用按钮并走同一取消路径。
- 主要弹窗补齐最小尺寸和统一最小尺寸提示，包括进度窗口、扫描四选项、修复完成详情、cleanup candidate、相似图列表、相似图组内对比、修复准备和设置窗口。
- 修复完成详情窗口修正内部伸缩权重，列表/详情区域滚动，底部关闭按钮固定可达。
